## 一、总体策略

### 1.1 核心理念

每一轮侧重一个领域，从产品可行性 → 多模态能力 → 对外变现与合规。

每一轮输出可交付文档集（PRD、架构、模块设计、接口示例、测试策略）。

**不在此阶段写代码**，仅规划与产出文档、POC 设计与评审材料。

### 1.2 版本命名与节奏建议

| 版本 | 可交付文档周期 | 侧重领域 | 时间目标 |
|------|------------------|----------|----------|

## 五、评审与验收流程

### 5.1 评审流程

**角色与职责**：

| 角色 | 职责 | 参与阶段 |
|------|------|----------|
| 产品经理 | PRD 评审、验收标准确认 | 所有版本 |
| 技术架构师 | 架构评审、技术选型确认 | v0.1+ |
| 后端开发 | 模块设计评审、API 规范确认 | v0.2+ |
| 测试工程师 | 测试计划评审、用例评审 | v1.0 |
| 安全专家 | 合规评审、隐私方案确认 | v1.0 |

**评审流程**：

1. **文档提交**：作者提交文档 PR
2. **评审准备**：评审者审阅文档
3. **评审会议**：集中讨论评审意见
4. **意见收集**：记录所有反馈意见
5. **修订迭代**：作者根据反馈修订文档
6. **再次评审**：修订后再次评审

**评审周期**：

| 版本 | 大改次数 | 小改次数 | 总周期 |
|------|----------|----------|--------|
| v0.1 | 1次 | 2次 | 3 天 |
| v0.2 | 1次 | 2次 | 5 天 |
| v1.0 | 1次 | 2次 | 8 天 |

### 5.2 验收流程

**验收标准**：

| 交付物 | 验收标准 | 验收方式 |
|--------|----------|----------|
| PRD 文档 | 内容完整、格式规范 | 文档评审 |
| Schema | 符合 JSON Schema 标准 | Schema 验证测试 |
| 架构草图 | 核心模块关系清晰 | 架构评审 |
| 用户调研 | 3+ 竞品、5+ 痛点分析 | 评审通过 |

**验收结果**：

- ✅ **通过**：所有验收项达标，可进入下一阶段
- ⚠️ **条件通过**：有少量待修复项，可在下一阶段前修复
- ❌ **不通过**：有重大问题，需要重新评审

## 六、沟通与审阅建议

### 6.1 Issues + PR 管理方式

**仓库选择**：DaimaRuge/Recipe-Agent（主仓库）

**Issue 管理策略**：

- 每个 Issue 对应一个文档或一个评审轮次
- Issue 标签：
  - `prd`: 产品需求文档
  - `architecture`: 架构设计
  - `module-design`: 模块设计
  - `tech-research`: 技术调研
  - `review`: 评审 Issue
  - `milestone`: 里程碑

**PR 管理策略**：

- 每个 PR 对应一个文档版本
- PR 标题格式：`docs(prd): v0.1 → v0.2`
- PR 描述必须包含：
  - 变更摘要
  - 讨论要点
  - 关联 Issue
  - 验收结果

### 6.2 协同流程

**版本提交流程**：

1. **文档草稿**：作者完成文档草稿
2. **内部评审**：团队内部审阅
3. **PR 提交**：提交 PR 到 Recipe-Agent
4. **评审反馈**：收集评审意见
5. **文档修订**：根据反馈修订文档
6. **再次评审**：修订后再次评审
7. **验收确认**：确认验收通过
8. **主仓库同步**：合并到 Recipe-Agent 主仓库

**沟通渠道**：

| 沟通事项 | 渠道 | 响应时间 |
|----------|------|----------|
| 紧急问题 | 飞书/微信/钉钉 | 2h 内 |
| 日常沟通 | GitHub Issues | 24h 内 |
| 评审会议 | 线上会议 | 按计划 |
| 进度同步 | GitHub Project Board | 每周 |
