## 总体策略

每一轮侧重一个领域，从产品可行性→多模态能力→对外变现与合规。每一轮输出可交付文档集（PRD、架构、模块设计、接口示例、测试策略）。不在此阶段写代码，仅规划与产出文档、POC 设计与评审材料。

### 三轮迭代核心聚焦

| 轮次 | 核心领域 | 主要目标 |
|------|----------|----------|
| **第一轮（v0.1）** | 产品与工程可行性 | 验证产品价值，完成基础文档 |
| **第二轮（v0.2）** | 多模态能力细化 | 深入技术细节，完成核心模块设计 |
| **第三轮（v1.0）** | 工程实施准备与合规 | 准备好开发，完成所有技术文档 |

#### 3. 产品/用户调研摘要

**竞品分析：**
- **下厨房：** 社区强，菜谱全，但缺乏个性化推荐和多模态能力
- **美食杰：** 内容丰富，但交互体验一般
- **豆果美食：** 社区活跃，但营养分析较弱

**用户痛点：**
- 不知道今天吃什么，决策困难
- 菜谱与实际情况不符（食材不全、口味不合适）
- 特殊需求（糖尿病、过敏等）难以满足
- 想做视频内容但缺乏专业工具

## 评审与验收流程

### 基本流程

1. **提交版本：** 我提交版本文档
2. **集中反馈：** 你评审并给出一次集中反馈（评论/标注）
3. **迭代修订：** 我根据反馈出下一版本

### 修改次数规则

- **每轮允许：** 一次大改 + 两次小修订（含讨论）
- **目标：** 在三轮内将 PRD/架构固化至可驱动 scaffold 的程度
- **限制：** 不直接提交代码，除非你授权

### 验收标准

每个版本的验收需要满足：
- [ ] 所有交付物齐全
- [ ] 文档质量符合要求
- [ ] 反馈意见已处理
- [ ] 双方确认无重大问题

## 沟通与审阅建议

### 文档版本管理

- **仓库：** DaimaRuge/chishenme3（待创建）
- **方式：** 以 Issues + PR 的方式管理文档版本
- **时机：** 我会按你允许时创建并提交

### PR 规范

每个文档 PR 应包含：
- **变更摘要：** 简要说明本次变更的主要内容
- **讨论要点：** 列出需要重点讨论的问题
- **关联 Issue：** 关联相关的 Issue（如有）

### PR 模板示例

```markdown
## 变更摘要
简要说明本次变更的主要内容...

## 讨论要点
1. 关于 X 的设计是否合理？
2. Y 模块的优先级是否需要调整？
3. ...

## 变更文件列表
- docs/prd_v0.2.md
- docs/module_design_v0.2.md
- ...

## 关联 Issue
#123
```
